# Briefing

What a fresh session needs to know about this project beyond what CLAUDE.md, ADRs, JTBD, and the voice guide already cover.

## What You Need to Know

- **The Shift is Windy Road's weekly AI newsletter.** Pipeline: `/wr-newsletter` skill. Substrate: `docs/ai-engineering-brief/ai-landscape.owm` (Wardley map, internal only, never named in the reader body). Draft output: `src/newsletters/drafts/<persona>/YYYY-MM-DD.md`. Persona split: `leader/` (The Shift) and `developer/` (Tokens Spent). Published editions move to `src/newsletters/published/<persona>/`. Logo: `src/newsletters/assets/the-shift-logo.{svg,png,-300.png}` (done, reused every week).
- **Review gate order is critic -> voice -> content-risk** (ADR 016). Critic runs first because analytical quality is expensive to fix; voice and content-risk are cheaper polish. Each critic round is a separate subagent call (fresh context) to avoid confirmation bias. Max three rounds.
- **Per-item voice capture is how the opener is built.** SKILL step 10 uses `AskUserQuestion` per candidate (Agree / Adjust / Drop). Tom's Adjust text feeds the drafter; the "From Tom" opener is assembled from the strongest POV across the week's adjusts, not from a model guess. Capture transcript persists at `<draft-folder>/YYYY-MM-DD.capture.md` (ADR 019, lands 2026-04-25 P015 verifying); step 11 drafter rule preserves Tom's load-bearing noun-phrases verbatim.
- **Logo is reused.** The SVG is done. Do not rebuild the logo each week; only the draft body changes.
- **Every weekly issue gets a unique title.** "The Shift" is the series; individual editions each get a POV-carrying H1 (example: `Container wars again: agent runtimes just became default infrastructure`). Date goes in the subtitle line.
- **Link sources inline on claim text, not in a Source block.** LinkedIn renders inline Markdown links; a trailing URL block reads as a wall of raw text. Enforced deterministically by `scripts/check-newsletter-structure.sh` (P089) at `/wr-newsletter` step 16 save.
- **This website repo's TDD runner is vitest, NOT bats.** `package.json` `test` is `vitest run`; there is no bats harness here (bats is the separate `packages/itil` plugin's convention, and this repo has no `packages/`). To behaviourally test a shell script, add a sibling `<name>.test.mjs` that execs it via `node:child_process` and asserts on exit code + stderr. Precedents: `scripts/render-cover.test.mjs` (imports a `.mjs`), `scripts/check-newsletter-structure.test.mjs` (execs a `.sh` via `spawnSync('bash', ...)`). Resolve script paths via `process.cwd()` (vitest runs from repo root); `import.meta.url` is not a usable file URL under vitest's transform.
- **The first part of a new session is stale-marker territory.** Session-start hooks block edits until architect/JTBD/style/voice/a11y subagents run. Any new `.claude/agents/*.md` file is not discoverable until Claude Code session restart; fall back to `general-purpose` with the subagent's behaviour inlined until the next session.
- **To re-satisfy a gate marker you must use a FRESH `Agent` tool call, not a `SendMessage` resume.** The architect/JTBD/etc unlock markers are written by a `PostToolUse:Agent` hook (`architect-mark-reviewed.sh` parses the agent's `**Architecture Review: PASS**` heading). A `SendMessage` resume of an already-spawned subagent runs the agent and produces the verdict text, but it is NOT an `Agent` tool call, so the PostToolUse hook never fires and the marker never lands; the edit stays blocked. When an architect/gate review returns `ISSUES FOUND` (mechanical FAIL, no marker) and you resolve the issues, re-run the gate as a brand-new `Agent(subagent_type: 'wr-architect:agent')` call so the PASS heading lands the marker. Cost this session: one wasted SendMessage round-trip before re-dispatching as a fresh Agent call. Note the architect emits `ISSUES FOUND` even for advisory-only findings it tells you to "proceed" past, so a re-run is the normal path after honouring its constraint.
- **Problem tickets live in `docs/problems/<state>/NNN-slug.md`** (per-state subdirectory layout, ADR-031: `open/`, `closed/`, `known-error/`, `verifying/`, `parked/`). Highest existing is 088; start at 089 for new tickets. (The old flat `docs/problems/NNN-slug.status.md` suffix layout was migrated; some briefing references and ticket bodies still cite the flat form.)
- **`/wr-itil:transition-problem` only does single-step lifecycle moves** (Open to Known Error to Verifying to Closed); its validation table rejects anything else. A **decision-complete close** (a ticket resolved by a recorded direction decision with no fix released this close, e.g. "keep as-is / scope narrowed / superseded elsewhere") is an **Open-or-Known-Error to Closed bypass per ADR-079**. Do it via the manage-problem / review-problems Step 4.6 relevance-close mechanics: `git mv` to `closed/`, set Status to Closed, append a `## Closed as no longer relevant` audit section (evidence shapes: `ADR-shipped-confirmed`, `self-marker-in-body`), refresh README (drop from WSJF Rankings, add to Closed, rotate line 3 + archive prior to `README-history.md`), then commit through the risk gate. P071 closed this way 2026-06-15.

- **The Bash tool runs zsh, so bash-only array syntax fails with "bad substitution".** A scripted batch operation (the 2026-06-28 verification-queue drain) used a bash associative array with `declare -A` and `${!EV[@]}` inline in a Bash tool call; zsh rejected `${!EV[@]}` with `(eval):N: bad substitution` and nothing ran (which is the safe outcome: an aborted compound command leaves no partial state). For any non-trivial script using bash arrays, `${!arr[@]}`, `${arr[@]}` expansions, or other bashisms, write it to a file (e.g. `/tmp/foo.sh` with a `#!/usr/bin/env bash` shebang) and run it with `bash /tmp/foo.sh`, OR use zsh-native syntax (`typeset -A`, `${(k)arr}`). Do not assume the heredoc/inline shell is bash.

- **A stray `.env` FIFO makes `next build` (and `next dev`) hang forever at 0% CPU.** P001 (open since April, "build hangs locally") was root-caused 2026-06-27 to `./.env` being a named pipe (FIFO, `prw-------`), not a regular file. Next loads `.env` files at the very start of the build; opening a FIFO read-only blocks indefinitely under POSIX (it waits for a writer that never comes), so the build hangs right after the prebuild step with no output and no CPU use. Diagnosis path that works: `sample <pid>` the stuck `node .bin/next build` process (it shows a blocked `open()` / `uv_fs_open` syscall), then `ls -la .env` (look for the `p` FIFO prefix) or `find . -type p`. Fix: `rm -f .env`. It is gitignored so it never reaches CI (which is why windyroad.com.au stayed live the whole time); the real local env lives in `.env.local`, and the build needs no custom env vars. The earlier 2026-05-30 "unreproducible" note checked process state rather than the `.env` file type, which is why it missed the cause for two months.
