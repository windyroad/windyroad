## Blocking findings

### B-01 — Activation sequencing is circular

The manifest makes separate queue/hash generation and runner/analysis implementation activation gates ([study.json](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/study.json:107)), while the preregistration says no Ollama queue or hashes exist until activation gates pass ([preregistration-v2.md](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/preregistration-v2.md:199)) and the README withholds “runtime code” until those gates pass ([README.md](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/README.md:27)).

Define an ordered state machine:

`permission → no-prompt preflight → fixture-only implementation and offline artifact generation → targeted review → OSF registration/amendment and freeze → outcome-call authorization`

Written permission must cover the automated preflight itself because Ollama's Terms prohibit automated access without permission. [Ollama Terms](https://ollama.com/terms)

### B-02 — The required review packet contradicts the amendment

The review protocol includes the manuscript and requires cross-document consistency ([independent-review.md](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/independent-review.md:9), [independent-review.md](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/independent-review.md:96)). It omits the conditional stratum, permission gate, separate analysis, structured-output difference, and possible new fixed subscription.

The prior approval covers `4b48101`, not this amendment. The old resolution statement that OSF registration may proceed is now stale and should be explicitly scoped or superseded.

## Major findings

### M-01 — The financial guard is incomplete

The amendment allows fixed subscriptions “newly purchased for this study” without a fixed-fee ceiling or explicit purchase authorization ([preregistration-v2.md](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/preregistration-v2.md:102)).

More importantly, Ollama documents that Pro/Max usage automatically draws from an existing extra-usage balance after included quota is exhausted. The activation preflight does not require proving that this balance is zero or disabled, and it is not repeated before each batch. [Ollama pricing](https://ollama.com/pricing)

Require an explicit fixed-fee authorization and a recorded zero/disabled extra-usage balance—or written provider assurance—before every batch and resume.

### M-02 — Authentication and drift controls are one-time and underspecified

“Full catalog digest” has no defined endpoint, canonicalization method, stored hash field, or repeat frequency. The protocol does not require rechecking Ollama account identity, plan, entitlement, server/client versions, exact candidate availability, catalog digest, or billing state before each batch.

Freeze a canonical preflight record and compare it on every resume. Accept a response only when its returned model exactly matches the frozen identifier; any mismatch must remain a durable, non-consuming suspension.

### M-03 — Analysis isolation is not implemented

[analyse.mjs](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/analyse.mjs:338) accepts unrestricted outcomes, omits model from its confirmatory aggregation key, and analyzes every supplied row. Balanced Ollama rows could therefore be pooled into confirmatory estimates. The function also always emits H1/H2 `supported` decisions, conflicting with the amendment's descriptive-only Ollama rule ([preregistration-v2.md](/Users/tomhoward/Projects/windyroad/research/llm-review-sequences/preregistration-v2.md:149)).

Before activation, add:

- An exact two-system allowlist that rejects missing, extra, or mixed confirmatory systems.
- A dedicated Ollama-only descriptive path with no `supported`, equivalence, superiority, or pooled output.
- Contamination-rejection and separate-ledger regression tests.

### M-04 — The future request contract is not fully frozen

The amendment correctly records that Ollama Cloud currently lacks structured-output enforcement and treats invalid JSON as an abstention. [Ollama structured-output documentation](https://docs.ollama.com/capabilities/structured-outputs)

The eventual request must also freeze and test the exact body: `stream: false`, tools disabled, explicit handling of `think`, no unsupported `format` schema, and exact parsing of the complete `message.content` without repair or fence stripping. Request-body and response-envelope hashes/schema versions should enter the separate authorization record.
